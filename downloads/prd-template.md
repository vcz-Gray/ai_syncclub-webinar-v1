# 4문항 PRD 템플릿

> PRD는 길게 쓰는 문서가 아닙니다. AI에게 보여줄 설계도입니다.
> 아래 4문항에 답하면, 그것이 곧 PRD 초안입니다.

---

## 빈 템플릿 (복사해서 바로 쓰세요)

```markdown
# [서비스 이름]

## 1. 누가 쓰나?
- 핵심 사용자:
- 이 사람이 있는 상황:

## 2. 뭐가 불편한가?
- 지금 겪는 문제:
- 왜 기존 방식이 불편한가:

## 3. 핵심 기능 3개 (P0만)
1.
2.
3.

## 4. 화면은 몇 개인가?
- 총 화면 수:
- 각 화면 이름과 목적:
  - 화면1: [이름] — [목적을 한 줄로]
  - 화면2: [이름] — [목적을 한 줄로]
  - 화면3: [이름] — [목적을 한 줄로]

## 한 줄 설명
이 서비스는 ___를 위해 ___를 더 쉽게 해준다.

## 성공 기준 (선택)
- 사용자가 ___분 안에 ___를 완료할 수 있다.

## 가정과 리스크 (선택)
- 가정:
- 리스크:
```

---

## 작성 원칙

| 원칙 | 설명 |
|------|------|
| **"왜"를 먼저, "무엇"을 나중에** | 기술 용어 없이 사용자 행동 중심으로 기술 |
| **P0만 남겨라** | P0(반드시 필요) 3개만. P1(나중 가능)은 별도 메모 |
| **3단계 워크플로우** | "사용자가 X → 시스템이 처리 → 사용자가 Y를 본다" |
| **구체적 숫자** | "빠르게" 대신 "2분 안에 첫 작업 완료" |
| **입출력 예시 포함** | AI에게 보여줄 때 예시가 있으면 품질이 대폭 향상 |

## 예시 1: 스터디 출석 체크 도구

```markdown
# 스터디 출석 체크 도구

## 1. 누가 쓰나?
- 핵심 사용자: 소규모(5~15명) 스터디 운영자
- 상황: 매주 출석 확인과 공지 전달을 카톡으로 하다가 기록이 유실됨

## 2. 뭐가 불편한가?
- 문제: 출석 체크와 공지가 매번 메신저에서 흩어진다
- 기존 방식: 카톡 투표, 구글 시트 → 흩어지고 누락됨

## 3. 핵심 기능 3개
1. 회차별 출석 체크 (토글 방식)
2. 공지 등록 및 기록
3. 참여자 관리 (초대/삭제)

## 4. 화면
- 총 4개
  - 홈 — 이번 회차 요약 + 출석률 차트
  - 출석 체크 — 참여자 목록 + 토글
  - 공지 기록 — 회차별 공지 목록
  - 참여자 관리 — 초대/삭제

## 한 줄 설명
스터디 운영자가 출석과 공지를 한 곳에서 관리하는 도구.

## 성공 기준
- 운영자가 1분 안에 출석 체크를 완료할 수 있다.
```

## 예시 2: 콘텐츠 검수 대시보드

```markdown
# 콘텐츠 검수 대시보드

## 1. 누가 쓰나?
- 핵심 사용자: 3~5명 규모 콘텐츠팀 운영자
- 상황: 검수 요청이 슬랙, 메일, 노션에 흩어져 우선순위가 안 보임

## 2. 뭐가 불편한가?
- 문제: 어떤 콘텐츠가 급한지, 누가 담당인지 한눈에 안 보인다
- 기존 방식: 슬랙 DM + 노션 표 → 상태 업데이트가 안 됨

## 3. 핵심 기능 3개
1. 검수 요청 등록 (제목, 마감일, 담당자)
2. 상태 변경 (대기 → 진행 → 완료)
3. 담당자별 필터 + 마감 임박 강조

## 4. 화면
- 총 3개
  - 요청 목록 — 칸반 또는 테이블 뷰, 상태별 정렬
  - 요청 상세 — 사이드 패널, 히스토리
  - 통계 요약 — 처리율, 평균 소요 시간

## 한 줄 설명
콘텐츠 팀이 검수 요청을 접수하고 상태를 추적하는 내부 도구.
```

## 예시 3: 개인 루틴 기록 앱

```markdown
# 개인 루틴 기록 앱

## 1. 누가 쓰나?
- 핵심 사용자: 하루 루틴을 기록하고 싶은 직장인
- 상황: 매번 시작은 하는데 기록이 끊기고, 유지율이 안 보임

## 2. 뭐가 불편한가?
- 문제: 기록 자체가 번거럽고, 어떤 루틴이 유지되는지 피드백이 없다
- 기존 방식: 메모앱, 캘린더 → 꾸준히 쓰기 어려움

## 3. 핵심 기능 3개
1. 오늘 루틴 체크 (원탭 토글)
2. 주간 달성률 리포트 (원형 프로그레스)
3. 연속 달성일 표시 (동기부여)

## 4. 화면
- 총 3개
  - 오늘 루틴 — 체크리스트 + 프로그레스 바
  - 주간 리포트 — 달성률 차트 + 연속일
  - 설정 — 루틴 추가/수정/삭제
```

## 예시 4: 클래스 예약 서비스

```markdown
# 소규모 클래스 예약 서비스

## 1. 누가 쓰나?
- 핵심 사용자: 1인 클래스/워크숍 강사
- 상황: 수강생이 카톡 DM으로 신청해서 관리가 안 됨

## 2. 뭐가 불편한가?
- 문제: 누가 신청했는지, 잔여 좌석이 몇 개인지 추적이 어렵다
- 기존 방식: 카톡 + 구글 시트 → 실시간 반영 안 됨

## 3. 핵심 기능 3개
1. 클래스 목록 + 잔여 좌석 표시
2. 온라인 신청 폼
3. 신청 확인 + 운영자 관리 화면

## 4. 화면
- 총 4개
  - 랜딩 — 클래스 소개 + CTA
  - 클래스 목록 — 일정, 잔여석, 가격
  - 신청 폼 — 이름, 연락처, 클래스 선택
  - 완료 — 신청 확인 메시지
```

## 예시 5: 모임 일정 조율 서비스

```markdown
# 모임 일정 조율 서비스

## 1. 누가 쓰나?
- 핵심 사용자: 5~10명 소모임 운영자
- 상황: 매번 "언제 돼?" 투표를 카톡에서 반복함

## 2. 뭐가 불편한가?
- 문제: 날짜 후보를 모으고 결과를 정리하는 과정이 비효율적
- 기존 방식: 카톡 투표 → 결과 정리가 수동

## 3. 핵심 기능 3개
1. 날짜 후보 등록
2. 참여자 투표 (가능/불가능/미정)
3. 결과 자동 집계 + 확정

## 4. 화면
- 총 3개
  - 일정 생성 — 제목, 날짜 후보 입력
  - 투표 — 참여자별 가능 여부 체크
  - 결과 — 막대 그래프 + 확정 버튼

## 한 줄 설명
소모임 운영자가 날짜 조율을 카톡 없이 끝내는 도구.
```

## PRD를 AI에 넣기 전 체크리스트

- [ ] 핵심 사용자가 1명으로 특정되어 있는가?
- [ ] 핵심 문제가 한 문장으로 표현되는가?
- [ ] 기능이 3개 이하로 압축되어 있는가?
- [ ] 화면 수와 각 화면의 목적이 명시되어 있는가?
- [ ] "이 서비스는 ___를 위해 ___를 더 쉽게 해준다"가 완성되는가?
