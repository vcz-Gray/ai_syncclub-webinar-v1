개인 서비스 아이디어

바이브 코딩으로 빠르게 프로토타입할 수 있는 1인/소규모 서비스 예시입니다.

스터디 운영 도구

문제: 공지와 출석 관리가 메신저에 흩어진다.

PRD 초안 사용자: 스터디 운영자 / 문제: 출석과 공지 분산 / 기능: 회차별 출석, 공지 등록, 참여자 관리 / 화면: 4개
Stitch 입력 "스터디 운영 도구. 홈 대시보드, 출석 체크 화면, 공지 기록 페이지, 참여자 관리 화면. 밝은 배경, 둥근 카드, 모던한 느낌."
정제 프롬프트 "출석 체크를 토글 스위치로 바꾸고, 회차별 통계 차트를 홈에 추가해줘."

정제 프롬프트 "투표 현황을 가로 막대 그래프로 시각화하고, 확정 버튼에 강조색 적용해줘."

대시보드 / 관리자 도구

초기 프롬프트 "[서비스명] 관리자 대시보드. 좌측 사이드바 내비게이션, 상단 KPI 카드 4개, 메인 영역에 테이블 뷰. Linear/Notion 스타일, 다크 테마."
정제 순서 1) KPI 수치 구체화 → 2) 필터/검색 추가 → 3) 상태 색상 체계 적용 → 4) 반응형 조정

정제 순서 1) 카드 레이아웃 조정 → 2) 필터 UI 개선 → 3) 상세 페이지 갤러리 추가 → 4) 모바일 최적화

정제 순서 1) 게시물 카드에 좋아요/댓글 수 표시 → 2) 카테고리 탭 → 3) 프로필 아바타 → 4) 알림 드롭다운

정제 순서 1) 캘린더 UI 개선 → 2) 가능 시간대 시각화 → 3) 확인 이메일 미리보기 → 4) 모바일 스텝 위저드

랜딩 페이지

핵심: 히어로 → 기능 소개 → 소셜 프루프 → CTA 흐름.

초기 프롬프트 "[서비스명] 랜딩 페이지. 히어로(헤드라인+CTA), 기능 3개 소개, 사용자 후기, 가격표, 하단 CTA. 모던하고 전환율 높은 디자인."
정제 순서 1) 헤드라인 카피 강화 → 2) 기능 섹션에 아이콘/일러스트 → 3) 소셜 프루프 추가 → 4) CTA 색상 대비 강화

Prototypes 연결 패턴

핵심: 멀티스크린을 하나의 흐름으로 연결하기.

초기 프롬프트 "4개 화면을 같은 디자인 시스템으로 이어서 만들어줘: 홈 → 입력 → 결과 → 상세. 내비게이션 일관성 유지."
정제 순서 1) 화면 간 전환 버튼 확인 → 2) 내비게이션 일관성 → 3) Prototypes로 클릭 흐름 연결 → 4) 분기 흐름 추가

AI Studio Build 모드 프로젝트 예시

Stitch 다음 단계로, 실제 작동하는 앱을 만드는 AI Studio 활용 예시입니다.

레시피 검색기 (Gemini 연동)

재료를 입력하면 Gemini가 레시피를 추천하는 풀스택 앱.

Build 프롬프트 "재료 입력 → Gemini API로 레시피 추천 → 결과 카드 표시. 검색 히스토리 저장."
활용 포인트 Gemini API 자동 연동, 실시간 프리뷰에서 바로 테스트 가능

퀴즈 앱

주제별 퀴즈를 풀고 점수와 오답을 확인하는 앱.

Build 프롬프트 "퀴즈 앱. 카테고리 선택 → 문제 풀기(4지선다) → 결과 화면(점수+오답 해설). 게이미피케이션 느낌."
활용 포인트 Code 탭에서 퀴즈 데이터를 직접 수정, Cloud Run으로 즉시 배포

서베이 빌더

드래그앤드롭으로 설문을 만들고 결과를 시각화하는 도구.

Build 프롬프트 "설문 빌더. 문항 추가(객관식/주관식/별점), 미리보기, 응답 수집, 결과 차트. Google Forms 참고."
활용 포인트 Annotate로 폼 요소를 클릭해 직접 수정, GitHub 연동으로 버전 관리

팁 계산기

금액과 인원을 입력하면 1인당 금액을 계산하는 간단한 앱.

Build 프롬프트 "팁 계산기. 금액 입력, 팁 비율 슬라이더, 인원 수 입력, 1인당 금액 표시. 깔끔한 카드 UI."
활용 포인트 AI Studio 입문 프로젝트로 최적. 10분 내 완성 가능.

PRD 있을 때 vs 없을 때

같은 아이디어를 PRD 유무로 비교한 결과입니다. 웨비나 슬라이드 13번 내용의 확장판입니다.

PRD 없이 Stitch에 넣었을 때

  • 화면은 그럴듯하지만 사용자가 누군지 모호함
  • 기능이 과도하게 많이 생성됨 (8~12개)
  • 화면 간 흐름이 연결되지 않음
  • 수정 지시 기준이 없어 반복 수정 5~10회
  • 스타일이 화면마다 미묘하게 다름
  • 결과물이 "데모"에 가까움

PRD를 넣고 Stitch에 넣었을 때

  • 사용자와 문제가 선명하게 반영됨
  • 기능이 P0 3개에 집중됨
  • 화면 간 흐름이 자연스럽게 연결됨
  • 수정 1~3회로 실서비스 근접 품질 달성
  • 일관된 디자인 시스템 유지
  • 결과물이 "프로토타입"에 가까움

Q&A 즉석 대응 패턴

웨비나 Q&A에서 자주 나오는 질문과 대응법입니다.

"아이디어가 너무 큰 경우"

참가자: "앱 하나로 다 하고 싶은데요."

대응 핵심 사용자 1명, 핵심 문제 1개, 기능 3개만 남기세요. 나머지는 v2에서.
Stitch 가이드 "가장 중요한 첫 사용자 흐름 3개 화면만 보여줘"

"내부 도구를 먼저 만들고 싶은 경우"

참가자: "외부 서비스보다 팀 내부 도구가 급해요."

대응 내부 업무 흐름에서 가장 자주 반복되는 1개를 골라 시작하세요.
Stitch 가이드 "작은 팀 내부 도구 UI로, 관리자 대시보드처럼 단정하게 구성해줘"

"교육용 서비스를 생각하는 경우"

참가자: "스터디/강의 관련 서비스를 만들고 싶어요."

대응 학습자와 운영자 중 누구의 문제를 먼저 풀지 정하세요.
Stitch 가이드 "학습 흐름이 보이는 3개 화면을 우선 보여줘"

"디자인이 마음에 안 드는 경우"

참가자: "결과는 맞는데 분위기가 어색해요."

대응 문제는 기능이 아니라 시각적 톤입니다. 레퍼런스 앱을 제시하세요.
Stitch 가이드 "더 신뢰감 있고 실제 서비스 프로토타입처럼 다듬어줘. [앱이름] 참고."

"코딩을 전혀 모르는 경우"

참가자: "코딩을 전혀 모르는데 가능한가요?"

대응 Stitch는 코딩 지식 없이 사용 가능합니다. PRD 4문항만 작성하면 됩니다.
Stitch 가이드 한국어로 자연스럽게 설명하면 됩니다. 기술 용어 불필요.

"바이브 코딩의 한계가 궁금한 경우"

참가자: "이걸로 실제 서비스까지 만들 수 있나요?"

대응 프로토타입과 MVP까지는 충분합니다. 프로덕션 수준은 개발자 협업이 필요합니다.
핵심 Y Combinator W25 배치의 25%가 코드 95%를 AI로 생성. 점점 경계가 옅어지고 있습니다.
