바이브 코딩이 뭔지, 어떤 도구를 쓰는지, 어떻게 시작하는지를 다루는 책의 첫 두 챕터입니다. 웨비나에서 다룬 Stitch, PRD, AI Studio의 배경 지식이 모두 여기에 있습니다. 오늘 밤 읽기에 딱 좋은 분량입니다.

PDF 다운로드 (DAY 0~1)

오늘 바로 실행하는 5단계

실행 순서

  • 1단계: 웨비나 슬라이드를 다시 보며 내 서비스 유형을 하나 고릅니다.
  • 2단계: PRD 템플릿 4문항에 답합니다 — 누가 쓰나? 뭐가 불편한가? 핵심 기능 3개? 화면 몇 개?
  • 3단계: Stitch에 PRD를 붙여넣고 첫 화면 1개를 생성합니다.
  • 4단계: 추가 화면 2개를 더 만들고, Prototypes로 흐름을 연결합니다.
  • 5단계: 아래 "1일-3일-7일 후속 액션"을 따라 개선 루프를 시작합니다.

많이 막히는 지점과 해결법

  • 아이디어가 너무 크다 — 핵심 사용자 1명, 핵심 문제 1개로 축소하세요.
  • 기능이 너무 많다 — P0(반드시 필요) 3개만 남기고 나머지는 P1으로 미루세요.
  • 화면이 안 나온다 — 페이지 수를 먼저 지정하고, 각 페이지의 목적을 한 줄로 적으세요.
  • 결과가 엉뚱하다 — 레퍼런스 앱을 직접 언급하세요. "에어비앤비 검색 화면처럼" 같은 식으로.
  • 디자인이 촌스럽다 — "다크 테마, Material 3, 둥근 카드" 같은 스타일 키워드를 추가하세요.
  • 다음 단계가 막막하다 — 후속 액션 가이드를 그대로 따라가세요.

Google Stitch 프로 팁 7선

Stitch를 10배 더 잘 쓰는 실전 테크닉입니다.

맥락을 먼저 제공하라

"피트니스 앱"이 아니라 "바쁜 직장인을 위한 피트니스 앱 대시보드, 일일 걸음수/칼로리/그래프 표시"처럼 대상 사용자 + 화면 목적 + 핵심 데이터를 함께 적어야 좋은 결과물이 나옵니다.

스타일을 구체적으로 지정하라

"다크 테마, 네온 그린 강조색, 둥근 카드, 모던 폰트, Material 3 패턴" 같이 색상 모드, 강조색, 카드 형태, 디자인 시스템을 명시하면 원하는 방향에 훨씬 가깝게 생성됩니다.

참조 앱을 직접 언급하라

"에어비앤비 검색 인터페이스와 유사한 카드 기반 레이아웃"처럼 실존 앱의 특정 화면을 레퍼런스로 제시하면 AI가 패턴을 정확히 이해합니다.

이미지/스케치를 업로드하라

손으로 그린 와이어프레임이나 스크린샷을 업로드한 뒤 "이 스케치를 편집 가능한 UI로 변환해줘"라고 지시하면 빠르게 고품질 UI를 얻을 수 있습니다.

Standard vs Experimental 모드

Standard 모드(Gemini 2.5 Flash, 월 350회)는 빠른 아이디어 검증용. Experimental 모드(Gemini 2.5 Pro, 월 50회)는 이미지 기반 정밀 디자인용. 회수가 제한되니 전략적으로 배분하세요.

채팅으로 부분 수정하라

"CTA 버튼 크기 키우고 내비게이션을 하단으로 이동"처럼 대화형으로 반복 개선하는 것이 처음부터 다시 생성하는 것보다 훨씬 효율적입니다.

코드 내보내기 전 패턴을 명시하라

내보내기 전에 "production-ready HTML/CSS"를 명시하고, 내비게이션, 폼 같은 표준 컴포넌트를 사용하면 깨끗하고 반응형인 코드를 얻습니다.

AI 시대의 PRD 작성 베스트 프랙티스

AI 도구에 넣을 때 최고의 결과를 내는 PRD 작성법입니다.

작성 원칙

  • "왜(Why)"를 먼저, "무엇(What)"을 나중에 — 기술 용어 없이 사용자 행동 중심으로 기술하면 AI가 더 정확한 결과를 만듭니다.
  • P0/P1/P2 우선순위를 반드시 매겨라 — AI가 한 번에 너무 많은 기능을 만들려다 품질이 떨어지는 것을 방지합니다.
  • 3단계 워크플로우로 적어라 — "사용자가 X를 한다 → 시스템이 처리 → 사용자가 Y를 본다" 형식이 AI에게 가장 명확합니다.
  • 성공 지표를 구체적 숫자로 — "사용자가 2분 안에 첫 작업을 완료" 같은 측정 가능한 기준이 프로토타입 품질 판단 기준이 됩니다.

실전 체크리스트

  • 핵심 사용자가 1명으로 특정되어 있는가?
  • 핵심 문제가 한 문장으로 표현되는가?
  • 기능이 3개 이하로 압축되어 있는가?
  • 화면 수와 각 화면의 목적이 명시되어 있는가?
  • AI가 자동 처리하는 부분과 사용자 승인이 필요한 부분이 구분되어 있는가?
  • 가정(Assumptions)과 리스크가 기록되어 있는가?
  • 입출력 예시가 2~3개 포함되어 있는가?

바이브 코딩 핵심 원칙

Andrej Karpathy가 제안한 바이브 코딩의 실전 규칙입니다.

한 번에 하나씩만

"전체 앱을 만들어줘"가 아니라 "로그인 화면 만들어줘" → 테스트 → "대시보드 만들어줘" → 테스트. 점진적으로 빌드해야 오류를 추적할 수 있습니다.

프롬프트 4요소 구조

맥락(Context) → 작업(Task) → 가이드라인(Guidelines) → 제약조건(Constraints). 이 4요소가 모두 있으면 생성 품질이 극적으로 올라갑니다.

빠르게 만들고 버려라

Karpathy의 핵심 철학: 코드는 "공짜이고 폐기 가능하다." 완벽한 코드를 만들려 하지 말고, 빠르게 만들고 → 테스트 → 필요하면 버리고 다시 만드는 사이클을 돌리세요.

반드시 결과를 검증하라

연구에 따르면 AI 생성 코드의 52%에 오류가 있습니다. "바이브"는 생성 과정이지, 검증을 건너뛴다는 뜻이 아닙니다.

큰 모델로 설계, 작은 모델로 실행

Claude나 GPT-4로 아키텍처를 계획한 뒤, 실제 코딩은 빠른 모델로 수행하는 하이브리드 전략이 비용과 품질을 모두 잡습니다.

도구 하나부터 마스터하라

Cursor, Stitch, Claude, AI Studio를 한꺼번에 쓰려 하면 어느 것도 제대로 못 씁니다. 하나를 먼저 충분히 익힌 뒤 다음 도구로 넘어가세요.

Google AI Studio Build 모드 활용법

Stitch 다음 단계로 넘어갈 때 알아두면 좋은 실전 팁입니다.

핵심 활용법

  • 단순하게 시작하고 대화로 발전시켜라 — "레시피 추천기"처럼 핵심 하나로 시작한 뒤 채팅으로 점진 확장.
  • Annotate 기능을 적극 활용하라 — 프리뷰에서 해당 요소를 직접 클릭한 뒤 수정 지시를 하면 AI가 정확히 이해합니다.
  • Code 뷰로 수동 수정을 병행하라 — 간단한 텍스트/스타일 변경은 코드에서 직접 수정이 더 빠릅니다.
  • ZIP/GitHub으로 내보내 후속 개발에 연결하라 — 생성된 코드는 표준 패턴이라 이어서 개발하기 좋습니다.

알아두면 좋은 것

  • 무료 티어는 분당 5~15회 요청 제한이 있습니다. 요청을 모아서 한 번에 하는 습관을 들이세요.
  • "I'm Feeling Lucky" 버튼으로 랜덤 아이디어를 받을 수 있습니다. 영감 출발점으로 유용합니다.
  • Chat 모드에서 프롬프트를 먼저 테스트한 뒤 Build 모드에서 통합하면 완성도가 올라갑니다.
  • 생성된 앱은 Cloud Run으로 바로 배포할 수 있지만, 에러 핸들링과 보안 강화는 추가로 필요합니다.

초보자가 꼭 피해야 할 8가지 실수

AI 프로토타이핑을 시작할 때 가장 자주 빠지는 함정들입니다.

처음부터 완벽하게 만들려 한다

프로토타입의 목적은 아이디어를 검증하는 것입니다. 색상/폰트/정렬 전에 핵심 기능이 작동하는지부터 확인하세요.

첫 번째 아이디어에 집착한다

AI의 장점은 빠르게 여러 버전을 만들 수 있다는 것입니다. 최소 2~3개 변형을 만들어 비교하세요.

AI 출력을 검증 없이 사용한다

AI 생성 코드의 52%에 오류가 있다는 연구 결과가 있습니다. 반드시 직접 테스트하세요.

도구를 너무 많이 동시에 배운다

한꺼번에 여러 도구를 쓰면 어느 것도 제대로 못 씁니다. 하나부터 마스터하세요.

"데모"만 만들고 "제품"을 안 만든다

인상적인 시연용이 아니라, "사용자가 매일 어떻게 쓸까?"를 항상 질문하세요.

모델 교체가 답이라 생각한다

프롬프트 품질과 요구사항 정의가 모델 선택보다 결과에 훨씬 큰 영향을 미칩니다.

첫 실패 후 포기한다

프롬프트를 개선하면 결과가 극적으로 달라집니다. 최소 3~5번은 바꿔가며 시도하세요.

기본기를 완전히 무시한다

HTML/CSS 기본 구조 정도는 이해해야 AI 결과물을 판단하고 수정 지시를 내릴 수 있습니다.
